# 小游戏

使用ebiten引擎编写，通过wasm嵌入网页：

- 使用go语言编写游戏，开发期间不需要额外环境
    - 游戏资源应尽可能同步打包进二进制文件，而不是从目录读取，以避免正式环境的资源文件路径问题
- 游戏功能开发完成后，为游戏加入`postMessage`
    - 通过该方法向网页传递消息（例如上传一局游戏的成绩）
    - 网页端编写响应代码，再拼上当前登录信息，上传服务器。
- 使用go将游戏编译成wasm格式，编写html文件挂载游戏
    - html需要引入`wasm_exec.js`（在`$go/lib/wasm/`路径下），需要注意的是，该文件版本应与编译游戏所用go版本一致
- 在ui中通过iframe嵌入上一步编写的html文件
    - postMessage能够穿透html文件
- （wasm文件托管，可选）最终生成的wasm文件可能相对较大（例如本项目中的flip.wasm就有18m，即使通过压缩也要下载5m的内容），
  如果你的服务器网络带宽不足（例如我的服务器，下载这5m需要40s），可以考虑将大文件托管到github pages或其他地方

一个网页端游戏的工程文件包含：`flip.html`/`flip.wasm`/`wasm_exec.js`（参考`compile_game.sh`脚本）

- iframe: `<iframe src="flip.html" />`
- flip.html: `import { flip.wasm, wasm_exec.js }`

## ebiten编程

程序从运行一个`ebiten.Game`接口类型实例开始，周期性的调用对应接口方法：

- `Update() error`：你可以假设该函数每秒调用TPS次（默认60），首次画图前会调用该函数，即你可以在该函数内初始化状态信息
- `Draw(screen *Image)`：调用次数取决于用户环境，尤其是显示器刷新率
- `Layout(outsideWidth, outsideHeight int) (screenWidth, screenHeight int)`：定义窗口尺寸

整个画布坐标原点在左上角，横/纵坐标向右/向下增大

示例项目中二级结构虽然也定义了这些方法，但是程序不会去调它们，换句话说，二级结构的方法名和方法声明（输入输出参数）没有强制要求

- 二级结构：我们将程序周期性调用的`ebiten.Game`接口类型实例称为一级结构，一级结构可根据需要封装二级结构

## 个人经验

层级抽象：

- 一级结构：`ebiten.Game`接口类型实例，一方面给ebiten引擎调用，一方面组合全局状态管理、输入监控和其他二级结构
- 二级结构：
    - 输入监控：获取用户输入，考虑全局状态
    - 其他二级结构：在flip中，我们将窗口主体抽象成了二级结构body，它由三个板块组成，每个板块分别执行各自的输入处理和页面展示

构建标签：使用构建标签可以同时支持本地开发（go环境）和web部署（wasm）

- 参考`send_score_go.go`/`send_score_wasm.go`
- `syscall/js`包只能在编译为WebAssembly时使用

状态函数内带有状态：在函数中编写一个处理逻辑，除了这个逻辑本身，还会因为周期性调用，将函数分成两个时刻。

- 以flip card update函数为例，这个函数应判断当前是否点击卡牌以及点击卡牌是否有效，
  而一旦写完判断逻辑，这之前的代码**每次调用都生效**，这之后的代码则**只有点击了有效的卡牌才生效**。
  具体的，这个函数还做两件事：**计算卡牌翻转动画角度**以及**如果已经翻转两张卡牌，比较它们**，
  那么计算角度就要写在判断前面，因为我们希望翻转动画是连续的、不受鼠标事件影响的；
  而卡牌比较则要写在后面，因为我们希望在翻开第三张卡牌的时候，才将不同的前两张卡牌盖回去
  （如果写在前面，那就变成了*翻转动画结束后立刻将前两张卡牌盖回去（卡牌不同的场景下）*，而这时玩家可能还没看清楚第二张卡牌）。
- 基于这一点，状态控制函数的编写需要更细致的思考，而且不能像其他函数一样，简单的根据代码逻辑就优化写法，
  因为即使是将两个分别操作不同内容的函数调换顺序都有可能产生意想不到的问题
