# 设计

*归档*项目各项设计，以及为什么这样设计。

## 约定

- 程序中的时间均采用`毫秒级时间戳`
- 程序中的`[]byte`在传输、保存等场景中，均使用**16进制编码**，以保证其可读性。场景举例：
    - go json marshal会对`[]byte`进行base64编码，而dart不会
    - gorm在保存不可读的`[]byte`字段时，会报错
- 为复杂功能编写功能测试，为项目编写集成测试

## Web UI

`web`整体采用响应式设计：使用媒体查询当前屏幕宽度，应用不同的html、css

- 因为web只是服务端程序配套的展示页面，所以为了偷点懒，我们在手机上删除了一批页面，这样可以少做一点适配（手机端没有个人中心）
    - 具体的，移动端不提供个人中心页面的入口，并在路由守卫中要求登录，实现禁止通过修改url的方式访问
- html实现方式：`listen`屏幕宽度并保存到`pinia`，使用`pinia`+`v-if`决定展示哪一部分的html
- css实现方式：`@media (min-width: 1280px) {}`

## 小游戏

使用ebiten引擎编写，通过webAssembly嵌入网页。

- 使用go语言编写游戏，开发期间不需要额外环境
    - 游戏资源应尽可能同步打包进二进制文件，而不是从目录读取，以避免正式环境的资源文件路径问题
- 游戏功能开发完成后，为游戏加入`postMessage`
    - 通过该方法向网页传递消息（例如上传一局游戏的成绩）
    - 网页端编写响应代码，再拼上当前登录信息，上传服务器。
- 使用go将游戏编译成wasm格式，编写html文件挂载游戏
    - html需要引入`wasm_exec.js`（在`$go/lib/wasm/`路径下），需要注意的是，应使用编译游戏所用的go版本的该文件
- 在ui中通过iframe嵌入上一步编写的html文件
    - postMessage能够穿透html文件
- （wasm文件托管，可选）最终生成的wasm文件可能相对较大（例如本项目中的flip.wasm就有18m，即使通过压缩也要下载5m的内容），
  如果你的服务器网络带宽不足（例如我的服务器，下载这5m需要40s），可以考虑将大文件托管到github pages或其他地方

## 接口访问令牌

服务端签发无状态的`access token`，供客户端后续需要登录的接口使用

- 使用自定义请求头`Unnamed-Plan-Access-Token`传递`token`，客户端和服务端都是
- `token`结构：`hex({"user_name":[xxx]...}).hex(hash(payload, key))`
    - hash algorithm：`hmac-sha256`
    - payload: `hex({"user_name":[xxx]...})`, `.`之前的16进制字符串
    - key：启动时随机生成（重启服务会失效所有历史令牌）
    - 程序从`token`中解析`user name`并使用，通常不需要客户端传递
- (计划中)密钥轮换：将密钥版本号加入payload，同时维护两个版本的密钥，新密钥推出超过令牌有效期后，可以停止维护旧密钥

## 密码存储hash

命名：

- `password`: pwd from web
- `pwdSHA256`: `sha256('password text')`
- `key`: `argon2('pwdSHA256', 'salt')`
    - argon2算法派生的密钥（相同参数派生出的密钥相同）
- `pwdArgon2`: `encode(key)`
    - 编码后结构示意: `argon2id.v=19,m=65536,t=3,p=1.[saltHex].[keyHex]`
    - `salt`: 程序随机生成

密码存储不使用明文，前端传参数`pwdSHA256`、服务端保存的则是`pwdArgon2`

前端执行hash的目的是即使恶意攻击窃取了http请求参数，也无法恢复出密码明文，进而攻击者无法使用我们的ui；
后端执行hash的目的是即使恶意攻击窃取了数据库数据，也无法恢复出密码hash，进而无法调用接口。

## 数据库备份/恢复

我习惯于薅云服务器厂商的新人优惠，这些年常以17～29元/年的价格购买云服务器，好处是价格便宜，坏处就是需要换服务器。

本节备份/恢复功能就是为了应对这一情况。

备份：每天0点自动备份，也可以手动备份；备份完成会将数据库记录写成文件。
恢复：手动恢复，从文件中读取数据并写入数据库。

手动备份/恢复通过向程序发送信号实现：`pkill unnamed_plan_server -f -10`是备份，`-12`是恢复

软删除的数据也会备份

我们希望在备份/恢复功能模块中，禁用字段的更新时间自动追踪：

- 为什么不希望`UpdateAt`字段自动更新：
    - 字段含义冲突：如果备份会覆写该字段，则该字段将失去表示**用户更新时间**的意义
    - 逻辑错误：写数据库在设置备份时间之后，而如果为备份时间提供一定的冗余（例如设置备份时间为一个小时以后），又会导致冗余时间内的更新无法备份
- 备份环节的做法：使用`UpdateColumns`方法，这是gorm提供的专门用来跳过hook和更新时间追踪的方法
- 恢复环节的做法：`create on conflict`
    - 一条数据可能是全新的，也可能是备份过之后又有更新，所以在不在数据库都有可能
    - update在where子句匹配不到记录时会直接返回，而我们希望它创建一条新记录
    - create：如果在冲突时直接update all，更新时间还是会设置为当前时间，需要我们明确指出**使用我们提供的值更新**

文件结构：

- 在服务端程序同级创建文件夹`backup`/`recover`，文件夹内为每个表再建文件夹，一个可能的文件路径为：`backup/note/1.json`
- `recover`和`backup`的结构相同，实际应用中就是将`backup`下的内容复制到`recover`然后恢复

hash算法：将id每一个byte解析成uint8并相加，加和对16取余，得到的结果即为该条记录对应的文件名

## 测试

项目中有两种测试：一个是功能测试，一个是集成测试。

- 功能测试：保证函数正确执行，例如访问令牌的签发与验证、密码存储hash的计算与验证。功能测试代码通常和功能代码在一起
- 集成测试：保证接口正确运行，通过构造http请求的方式模拟正式使用场景。代码在单独目录`server/test/`

集成测试：模拟正式使用场景，需要启动服务端程序

- 使用测试数据库表：我们为测试创建测试表，表名为正式表名加前缀`t_`（gorm直接提供相关配置），让服务端程序使用测试表
- 在构造http请求之前还要做的事情：建表、修改序列的起始值、预设数据
    - 修改序列的起始值：现在不使用自增序列做id了，这一步省略
- 程序有多个接口、每个接口又可能有多个测试用例，我们为每个测试用例构造http请求，并假设它们的返回值：
  如果返回值（例如错误信息）或副作用（例如数据库改动）符合预期，则视为该用例测试通过；否则视为测试不通过。
- 如果所有用例测试通过，则删除测试表

## 问题

- 为什么采用16进制编码，而不是base32/64编码？  
  首先说明，我们会优先适配特殊算法使用的配套标准（例如totp密钥通常使用base32编码），而其他没有固定最佳实践的会使用16进制编码。
  简单介绍一下这几种编码方法：
    - 16进制编码每个字节包含4个有效bit位，结果长度为原始`[]byte`长度的200%；
    - base32编码每个字节包含5个有效bit位，原本5个字节的数据需要用8个字节表示，结果长度为原始长度的160%；
    - base64编码每个字节包含6个有效bit位，原本3个字节的数据需要用4个字节表示，结果长度为原始长度的133%；

  在现代编程中，除特殊情况外，这些长度都是可以接受的。接下来说明不选择base32/64编码的原因：
    - 字符集：两种编码都有多个字符集，而且base64的字符集是带特殊字符的，这意味着我们的业务逻辑会受到字符集特殊字符的限制
      （例如`-`属于base64url编码的字符集，假设我们使用base64编码密码hash，那么hash结构则不应包含`-`作为分隔符或其他）
    - 个人喜好：16进制更方便编程调试，其长度更容易换算到`[]byte`的长度
- 为什么数据库记录id使用uuid v7？
  数据库记录id比较常见的选择有：自增序列(sequence)和程序生成id。其中自增序列在分布式环境以及数据迁移场景中表现不佳
  （前者依赖中心化的id分配器；后者需要额外的序列设置，在业务逻辑依赖id时更需要格外注意），
  因为我的数据库可以预见的会进行多次数据迁移，所以我直接选择了程序生成id。
  选择uuid v7则是因为数据库底层存储时，无序的id会导致索引频繁分裂，所以选择相对有序的v7版本。
- 接口访问令牌的key在每次启动时随机生成，那么每次重新启动服务端程序都会使历史访问令牌失效，如何应对这一问题？
  尝试分片保存（例如配置文件、环境变量、数据库各持有key的一部分，然后使用固定密钥）(?)
