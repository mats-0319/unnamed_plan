# 设计

*归档*项目各项设计，以及为什么这样设计。

## 约定

- 程序中的时间均采用`毫秒级时间戳`
- 程序中的`[]byte`在传输、保存等场景中，均使用**16进制编码**，以保证其可读性。场景举例：
    - go json marshal会对`[]byte`进行base64编码，而dart不会
    - gorm在保存不可读的`[]byte`字段时，会报错
- 为复杂功能编写功能测试，为项目编写系统测试

## 目录结构

- internal：通用的http服务端代码，包括配置、数据库(DAO、ORM)、http、日志、统一的错误处理等，
  但实际上因为自用的原因，这一部分代码并没有那么通用，也就是说想要使用这部分代码创建新项目，不能直接用、还需要改改
- cmd：业务逻辑相关

## Web UI

`web`整体采用响应式设计：使用媒体查询当前屏幕宽度，应用不同的html、css

- 因为web只是服务端程序配套的展示页面，所以为了偷点懒，我们在手机上删除了一批页面，这样可以少做一点适配（手机端没有个人中心）
- html实现方式：`listen`屏幕宽度并保存到`pinia`，使用`pinia`+`v-if`决定展示哪一部分的html
- css实现方式：`@media (min-width: 1280px) {}`

## 接口访问令牌

服务端签发无状态的`access token`，供客户端后续需要登录的接口使用

- 使用自定义请求头`Unnamed-Plan-Access-Token`传递`token`
- `token`结构：`hex({"user_id":[xxx]...}).hex(hash(payload, key))`
    - hash algorithm：hmac-sha256
    - payload: `hex({"user_id":[xxx]...})`, `.`之前的16进制字符串
    - key：启动时随机生成
- 验证规则：
    - 结构：`str1.str2`
    - str2可由str1+hmac密钥计算获得
    - str1是16进制字符串，且能够反序列化到对应的结构
- (未实现)密钥轮换：将密钥版本号加入payload，同时维护两个版本的密钥，设置定时任务-删除老密钥，在密钥过期时间之后执行

## 密码存储hash

> 命名：
> - password: pwd from web
> - key: argon2算法派生的密钥，当作hash使用（相同参数派生出的hash密钥相同）
> - pwd hash: 指完整格式的、数据库实际保存的密码

密码存储不使用明文，界面上输入密码后执行一次hash放入请求体、服务端保存的则是前端密码加盐之后的hash

- 密码hash结构：`argon2id.v=19,m=65536,t=3,p=1.[saltHex].[keyHex]`
- web: `hex(hash('password'))`
- db: `encode(hash('pwd from web', 'salt'))`（这里的编码指按照特定顺序，组合算法参数、salt、key）
- hash algorithm: `sha256`(web) / `argon2id derive key`(server)
- 前端执行hash的目的是，即使恶意攻击窃取了http请求参数，也无法恢复出密码明文，进而攻击者无法使用我们的ui；
  后端执行hash的目的是，即使恶意攻击窃取了数据库数据，也无法恢复出密码hash，进而无法调用接口。
- 验证规则：解码`pwd hash`获得算法参数、salt、key，使用算法参数、salt和`password`计算new key，比较两个key

## 2FA验证(two-factor authenticate)

允许用户使用2FA验证的登录方式，默认不开启，可以通过在修改个人信息界面添加totp密钥的方式开启。

采用RFC 6238标准的TOTP算法（github同款）。

## 测试

项目中有两种测试：一个是功能测试，一个是接口测试。

- 功能测试：保证功能正确执行，例如访问令牌的签发与验证、密码存储hash的计算与验证。功能测试代码通常和功能代码在一起
- 接口测试：保证程序正确运行，通过构造http请求的方式模拟正式使用场景。代码在单独目录`server/test/`

接口测试：模拟正式使用场景，需要启动服务端程序

- 使用测试数据库表：我们为测试创建测试表，表名为正式表名加前缀`t_`（gorm直接提供相关配置），让服务端程序使用测试表
- 在构造http请求之前还要做的事情：建表、修改序列的起始值、预设数据
- 程序有很多接口、每个接口又可能有很多测试用例，我们为每个测试用例构造http请求，并假设它们的返回值：
  如果返回值（例如错误信息）或副作用（例如数据库改动）符合预期，则视为该用例测试通过；否则视为测试不通过。
- 如果所有用例测试通过，则删除测试表

## 问题

- 为什么采用16进制编码，而不是base32/64编码？  
  首先说明，我们会优先适配特殊算法使用的配套标准（例如totp密钥通常使用base32编码），而其他没有固定最佳实践的会使用16进制编码。
  简单介绍一下这几种编码方法：
    - 16进制编码每个字节包含4个有效bit位，结果长度为原始`[]byte`长度的200%；
    - base32编码每个字节包含5个有效bit位，原本5个字节的数据需要用8个字节表示，结果长度为原始长度的160%；
    - base64编码每个字节包含6个有效bit位，原本3个字节的数据需要用4个字节表示，结果长度为原始长度的133%；

  在现代编程中，除特殊情况外，这些长度都是可以接受的。接下来说明不选择base32/64编码的原因：
    - 字符集：两种编码都有多个字符集，而且base64的字符集是带特殊字符的，这意味着我们的业务逻辑会受到字符集特殊字符的限制
      （例如`-`属于base64url编码的字符集，假设我们使用base64编码密码hash，那么hash结构则不应包含`-`作为分隔符或其他）
    - 个人喜好：16进制更方便编程调试，其长度更容易换算到`[]byte`的长度
