# 设计

## 约定

- 程序中的时间均采用`毫秒级时间戳`
- 程序中的`[]byte`在传输、保存等场景中，均使用**16进制编码**，以保证其可读性。场景举例：
    - go json marshal会对`[]byte`进行base64编码，而dart不会
    - gorm在保存不可读的`[]byte`字段时，会报错
- 使用`apifox`模拟前端请求，用于在服务端开发过程中测试接口
    - 完成开发后发现对应的`apifox`接口描述文件相对较大，且无法**一键测试**，遂决定使用go发送http请求、编写测试

## 总体设计

- `web`整体采用响应式设计：使用媒体查询当前屏幕宽度，使用不同的html、css
    - 因为web只是服务端程序配套的展示页面，所以为了偷点懒，我们在手机上删除了一批页面，这样可以少做一点适配（手机端没有个人中心）
    - html实现方式：`listen`屏幕宽度并保存到`pinia`，使用`pinia`+`v-if`决定展示哪一部分的html
    - css实现方式：`@media (min-width: 1280px) {}`
- 接口访问令牌：服务端签发无状态的`token`，供客户端后续需要登录的接口调用
    - hash algorithm: `hmac-sha256`
    - 登录成功时，服务端签发`token`，后续前端请求则需要带上该`token`
    - 用户服务通过请求头返回`token`参数给网关，网关根据参数签发`token`
    - 因为服务端没有任何保存，所以无法限制用户在多终端登录
- 密码存储hash：密码存储不使用明文，界面上输入密码后执行一次hash放入请求体、服务端保存的则是密码hash加盐之后的hash
    - web: `hex(hash('password'))`
    - db: `hex(hash('pwd from web', 'salt'))`
    - hash algorithm: `sha256`(web) / `argon2id`(db)
    - 前端执行hash的目的是，即使恶意攻击窃取了http请求参数，也无法恢复出密码明文，进而攻击者无法使用我们的ui；
      后端执行hash的目的是，即使恶意攻击窃取了数据库数据，也无法恢复出密码hash，进而无法调用接口。

## 用户服务

- 只有登录功能返回token，前端创建用户成功之后自动调用一次登录

## 数据库备份（导入/导出）

应用场景：更换服务器时，数据库数据转移。出于学习目的，不使用数据库可视化工具的导入/导出功能。

流程设计：导出数据库数据，导入到另一个数据库

- 自动导出（例如24h）还是手动导出，或者两种都要？
- 导出的文件如何移动到另一个云服务器（下载+上传）
- 导入数据是否考虑目标数据库禁止远程访问？（如果考虑这一点，就得把读文件、导入数据库的代码放在云服务器执行）

Q：能不能整个过程都在一个地方完成，例如都在web端实现，而不是像导出使用web、上传使用ssh工具？
A：理论上可行：自动导出+手动导出 - 压缩下载 - 上传压缩包 - 解压缩并导入。使用gzip压缩工具

- 自动导出：见下节
- 手动导出：服务端与web建立SSE连接，同步导出进度
- 压缩下载：将导出文件夹压缩并下载到本地
- 上传压缩包：（此时在新的云服务器上部署的web ui）上传压缩包
    - 通过gorm dry run获得建表和预设管理员用户的sql，所以新服务器导入前还需要做一些事情：部署前后端代码、数据库执行sql
- 解压缩：解压缩，使用与导出不同的路径
- 导入：从文件读取数据，导入到数据库（要求已经建表且结构相同）

### 导出功能设计

为需要备份的表添加`exportedAt`字段

- `id`：数据库记录编号，uint类型的自增sequence（如果id不是数字类型自增序列，则定义一个这样的字段）
- `exportedAt`：上一次备份时间，如果该条记录在该时间后仍有修改(`exportedAt < updatedAt`)，则视为需要备份

方案概述：使用定时任务，例如每天搜索一次，将需要备份的数据都导出到文件

操作步骤：

- 查询要备份的数据表，条件是`exportedAt < updateAt`，分页查询，按照`exportID`升序排列
    - PageSize：100
- 根据`id`计算该条记录应该备份到的文件名
    - 文件名：`[table name]/data_[start id]to[end id].json`, e.g. `note/data_1to100.json`
- 打开目标文件，不存在则创建，读取文件内容并序列化到内存
    - 统一处理目标文件存在/不存在的情况
- 根据`id`是否匹配成功，修改文件中该条记录原本的值，或者新增一条记录；全部完成后将结果写入文件（覆盖式写入）
- 设置`exportedAt`为当前时间，写回数据库

## 测试

目前编写了模拟前端请求的测试，但是只验证了交互、没有验证数据状态（最后没有读数据库数据比较）

如果根据教科书上的测试标准，应该有：

- 功能测试，例如对登录功能、注册功能的测试，不需要启动服务端程序、模拟数据库连接
- 系统测试，就是我现在写的这种模拟前端请求的测试，它可以最完整的验证接口调用过程，需要启动服务端程序、数据库服务等配套环节

因为我们的每个功能都足够简单，不想写功能测试，计划只写系统测试，每次测试模拟前端调用、启动服务端程序、数据库服务。

虽然使用真实的数据库连接，但是测试数据和正式数据还是要分开的：

- 使用测试表，gorm连接配置里就有表名前缀的配置
- 为每个接口编写多个测试用例，保证流程的覆盖率，每个测试的结果都要验证：验证错误信息或者从数据库读数据对比
- 如果测试全部通过，删除测试表
